Alright, here’s the trick the big password managers use once basic autofill is working. It’s called a **learning matcher** or **adaptive domain mapping**. Sounds fancy, but the idea is simple: the manager **learns associations between apps/domains and vault entries based on the user’s behavior**, instead of relying only on static rules.

---

# 1. The problem this solves

Static domain matching works for most sites:

```
twitter.com → vault entry twitter.com
```

But real-world cases break this:

Examples:

```
company login portal
SSO systems
apps with strange domains
in-app WebViews
```

Example scenario:

User saved credential:

```
portal.company.com
```

But the login form appears in:

```
auth.companylogin.net
```

Your domain matcher will fail.

User then manually selects the vault entry.

If you do nothing, the same failure repeats forever.

---

# 2. Learning matcher concept

When the user manually selects a credential during autofill, the app records the association.

Example event:

```
Autofill request from:
auth.companylogin.net

User selected vault entry:
portal.company.com
```

System learns:

```
auth.companylogin.net → portal.company.com
```

Next time:

```
auth.companylogin.net
```

Your manager already knows which credential to suggest.

---

# 3. Data structure

Store a **learned mapping table**.

Example table:

```
learnedDomainMappings
```

Structure:

```
requestDomain
vaultDomain
confidence
timestamp
```

Example row:

```
auth.companylogin.net → portal.company.com
confidence: 0.9
```

---

# 4. How it works in autofill flow

Autofill request:

```
domain = auth.companylogin.net
```

Matching order:

```
1 exact match
2 subdomain match
3 app package mapping
4 learned mapping
```

If step 4 finds:

```
auth.companylogin.net → portal.company.com
```

You return those vault entries.

---

# 5. Increasing confidence

Don’t trust the first match immediately.

Increase confidence gradually.

Example:

```
manual match 1 → confidence 0.4
manual match 2 → confidence 0.7
manual match 3 → confidence 1.0
```

Only start auto-suggesting when confidence ≥ 0.7.

---

# 6. Expiry logic

Mappings should expire.

Example:

```
TTL = 30 days
```

Why?

Sites change login domains.

Expired entries get cleaned automatically.

---

# 7. Preventing phishing abuse

This part is critical.

Never learn mappings for **different root domains**.

Example:

```
amazon.com → good
evil-amazon-login.com → reject
```

Use Public Suffix List to verify root domain.

Safe learning cases:

```
login.company.com → company.com
auth.companylogin.net → companylogin.net
```

Unsafe case:

```
amazon.com → amazon-login.xyz
```

Reject.

---

# 8. UI transparency (important)

Users should be able to see learned mappings.

Settings page example:

```
Autofill Associations

auth.companylogin.net → portal.company.com
login.office365.com → microsoft.com
```

Allow:

```
Remove mapping
```

Otherwise users lose control.

---

# 9. Storage location

Store mappings in your **native vault database**.

Example table:

```
learned_domain_mappings
```

Columns:

```
request_domain
vault_domain
confidence
last_used
```

Index by `request_domain`.

---

# 10. Why this matters

Without this system, autofill reliability is maybe:

```
70–80%
```

With learning:

```
90–95%
```

This is why major managers feel “smart”.

They remember what the user did before.

---

# 11. Example real-world cases

These are real login mismatches password managers learn:

```
appleid.apple.com → apple.com
login.microsoftonline.com → microsoft.com
accounts.google.com → google.com
auth0.company.net → company.com
```

Your static matcher may not catch them.

Learning fixes it automatically.

---

# 12. Implementation complexity

This feature is actually **small code**:

```
~120–150 lines
```

But huge UX improvement.

Components:

```
Mapping table
Confidence update logic
Lookup during autofill
Settings UI
```

# 13. Where to add it in your plan

Add a new phase after your current bug fixes:

```
Phase 5 — Adaptive Domain Learning
```

Steps:

```
record manual selections
increase confidence
store mapping
use mapping during match
add management UI
```

---

# Final advice

Don’t implement this immediately.

First ship:

```
core autofill
domain matcher
multi-step login fix
heuristics
```

Once that is stable, add the learning system.

Otherwise debugging becomes impossible.
