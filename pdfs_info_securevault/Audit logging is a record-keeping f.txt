**Audit logging** is a record-keeping feature that tracks who did what and when inside your database. Think of it as a security camera for your data—it logs every action (like reading, writing, deleting, or even failed login attempts) along with the user or IP address that performed it, and the exact time.

### What gets logged?
- **Data access:** Who queried a password vault, when, and from where.
- **Schema changes:** If someone altered a table structure or added indexes.
- **Authentication attempts:** Successful and failed logins to the database itself.
- **Privilege changes:** If admin rights were granted or revoked.

### Why is it important?
- **Security investigations:** If a breach happens, audit logs help trace how it occurred and what data was accessed.
- **Compliance:** Regulations like GDPR, HIPAA, or SOC2 often require audit trails to prove data handling practices.
- **Troubleshooting:** Developers can see exactly what queries ran before an error occurred.

### Relevance to your password manager
For your app, **encryption happens at the application level**, so the database only stores encrypted blobs. Even if someone accessed the raw database, they’d only see ciphertext. However, audit logging would still be valuable for:
- Monitoring who (or what service) connected to the database—e.g., spotting an unexpected IP trying to query user data.
- Tracking administrative actions, like someone trying to drop tables or change security settings.

### Why the free tier lacking it is fine—for now
- **You are starting from zero:** At this stage, you likely don’t have compliance mandates or a large user base to monitor.
- **Alternative methods:** You can implement your own lightweight logging inside your application (e.g., log when users log in, change passwords, or delete vaults). That won’t give you database-level visibility, but it covers the most critical user actions.
- **Future upgrade:** If your app grows and you need formal audit trails, you can upgrade to a paid MongoDB Atlas tier or switch to a self-hosted solution where you can enable detailed logging yourself.

**Bottom line:** Audit logging is a nice-to-have for security and compliance, but it’s not a dealbreaker for a free-tier password manager MVP. Focus on building your core encryption features first.

### 🛡️ Addressing Your Concern: PBKDF2, AES, and Database Load

You asked specifically about using algorithms like **PBKDF2** and **AES-GCM** and whether that would be too heavy for a NoSQL database.

You do not need to worry about this. Here is the key distinction:

1.  **The Database is Just Storage**: In a properly designed password manager, the database (MongoDB) **never performs the encryption or hashing**. It simply stores the final output .
2.  **The App Does the Work**: The heavy computational work—like running PBKDF2 for thousands of iterations to derive a key, or encrypting data with AES—happens on the **client device** (or your backend server) **before** the data is ever sent to MongoDB .
3.  **MongoDB's Role**: By the time the data hits the database, it is just a string of random-looking text (ciphertext). For example, you might store an object that looks like this, where the `vault` data is already encrypted:
    ```json
    {
      "userId": "123",
      "vault": "U2FsdGVkX19OiEdh/Lkjg4Z6V...==", // <-- Already encrypted by your app
      "salt": "s78dfg67...", // <-- Random salt used for PBKDF2
      "iterations": 600000
    }
    ```
    MongoDB is exceptionally fast at storing and retrieving simple text blobs like this, so you won't experience a "heavy read/write" load.
