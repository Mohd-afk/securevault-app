**Feature Requirements & Bug Fixes – Password Manager App**  

This document consolidates the requested updates, bug fixes, and new features into a clear, actionable specification for implementation.

---

## 1. UI/UX Improvements

### 1.1 Forgot Master Password Placement  
- On the login screen, move the **"Forgot master password?"** link from below the email field to directly below the **Master Password input field**.  
- Increase the font size of the link slightly to improve visibility.

### 1.2 Password Card Layout – Prevent Horizontal Scroll  
- Fix the layout of the password detail screen so that content never exceeds the viewport width – eliminate any horizontal wobble or scrolling.  
- Ensure all elements inside the password card (including the **Share** button) are fully visible without being cut off. Adjust padding and spacing accordingly.

### 1.3 Category List – Expand/Collapse Functionality  
- In the main category list (e.g., "Others", "Phone"), add an **arrow up/down** icon at the end of each category row.  
- Clicking the **category name** toggles the expansion/collapse of the passwords under that category.  
- When expanded, show the saved passwords; when collapsed, hide them.

---

## 2. New Features & Behavior Changes

### 2.1 Soft Delete with Trash Bin  
- When a user deletes a saved password, move it to a **Trash bin** instead of permanently deleting it.  
- Enable an **auto-delete** mechanism: passwords in Trash are automatically and permanently deleted after **30 days**.  
- The auto-delete should be enabled by default and not require user intervention.

### 2.2 Native Share for Password Card  
- Replace the current copy‑to‑clipboard action on the **Share** button with a native share sheet.  
- The share sheet should display available apps (WhatsApp, Telegram, etc.) to share the password information.  
- Ensure the shared content includes the relevant password details (e.g., title, username, password) in a readable format.

### 2.3 Block Screenshots (Optional Toggle)  
- In **Settings**, add a toggle labelled **"Allow screenshots"** (default = ON).  
- When the toggle is OFF, prevent the user from taking screenshots inside the app (use `FLAG_SECURE` on Android or the equivalent on iOS).  
- If a screenshot attempt is detected while disabled, show a popup/toast: *"Screenshots are not allowed."*

### 2.4 Referral / Share App  
- In **Settings**, add a button **"Share with a friend"** or **"Refer the app"**.  
- Tapping it opens the native share sheet with a pre‑filled message containing the app download link.

### 2.5 In‑App Feedback  
- Add a **"Feedback"** button in Settings that opens a feedback form.  
- The form must include:  
  - **Title** (subject) – text input, required.  
  - **Note** (body) – multi‑line text input, required.  
  - **Attachment icon** – allows the user to attach one or more images.  
- On submission, send an email to **mohdjamal110@gmail.com** with:  
  - Subject = title entered by user.  
  - Body = note entered by user.  
  - Attachments = any images added.  
  - Sender information = include the user’s email address or user ID in the email body or as the reply‑to address.

## 3. Settings & Security Enhancements

### 3.1 Username Mandatory & Immutable  
- During account creation, **username** becomes a required field.  
- Once set, the username **cannot be changed** later (no edit option in profile/settings).

### 3.2 Password Reset via Email (Forgot Current Password)  
- In **Settings → Security**, add a **"Forgot current password?"** option.  
- When clicked:  
  1. Show a toast: *"Check your email for password reset instructions."*  
  2. Send an email to the user’s registered address containing a secure password reset link.  
  3. The link must open a dedicated page (in‑app or web) where the user can set a **new password**.  
  4. After successful reset, force the user to log in again with the new password.

### 3.3 Change Password – Enforce Complexity & Force Relogin  
- When changing the password (existing **Change password** feature), enforce the same complexity rules as during registration:  
  - Minimum 8 characters.  
  - At least one uppercase letter.  
  - At least one lowercase letter.  
  - At least one number.  
  - (Optional) at least one special character.  
- After a successful password change, automatically log the user out and redirect to the login screen.

### 3.4 Delete All User Data  
- In **Settings**, add a **"Delete all data"** button.  
- On tap, show a confirmation popup: *"All your data will be permanently deleted and cannot be recovered. Continue?"*  
- If user confirms, open a new dialog/page asking for the **master password**.  
- Upon correct master password entry:  
  - Delete all user data from the local database and backend.  
  - Immediately log the user out and return to the login screen.

---

## 4. Implementation Notes

- All changes must maintain existing functionality and follow the app’s current design system.  
- For Android, use `Intent.ACTION_SEND` for share sheets; for iOS, use `UIActivityViewController`.  
- Email sending for feedback can be implemented via an API endpoint that forwards the content to the specified email address, or by using a mail‑to link with limitations on attachments.  
- Screenshot blocking on Android requires setting `WindowManager.LayoutParams.FLAG_SECURE`; on iOS, use `UIApplication.shared.isIdleTimerDisabled` is not sufficient – consider `UITextField` secure entry or `UIScreen` capture detection.  
- The auto‑delete after 30 days should be implemented as a background job or check on app launch.
